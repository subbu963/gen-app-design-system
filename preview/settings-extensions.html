Worker extensions run in-webview behind a hardened Function scope. Deno extensions run as long-lived subprocesses with explicit --allow-* permission flags, a per-extension HTTPS proxy enforcing the manifest's network allowlist, and setrlimit(RLIMIT_AS,…) memory caps. MCP servers run as their own child processes. The harness sees one stream of tools.

Notes
  • Kind ≠ runtime. Kind chip carries authorship (bundled / user / npm · stdio / http · oauth); the runtime chip carries the sandbox (worker / deno). They're independent — a user extension can be either runtime.
  • Deno rows carry a process-state dot. 7px green pulse = running, gray = idle, red = crashed/backoff. Click the chevron to open the runtime detail sheet.
  • Trailing affordances vary by kind. User-authored gets a pencil; MCP gets a tools-list icon and a more menu; bundled gets nothing. Deno rows additionally get a chevron into the runtime detail.
  • "Re-auth needed" is a pill, not a banner. It rides next to the kind chip and replaces the toggle's color until resolved.
  • iOS soft-disables every deno row. Same pattern as the stdio row before — discoverable, not removed. Tap shows the iOS rationale block from the runtime detail.
